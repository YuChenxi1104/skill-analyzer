# Skill 流程图规范

## 1. 目标

`skill 流程图` 用于解释一个 Skill 如何从输入走到交付。
它回答的问题是：

`用户说完之后，接下来发生了什么？`

默认产物是 merged-preview.html 里的完整流程图模块，并作为中级模块里的 `2.2 流程图` 出现。

## 2. 边界

### 它是什么

- 执行过程视图
- 从输入到交付的过程拆解
- 面向学习和讲解的流程图

### 它不是什么

- 结构图翻版
- 文件目录图
- 教学海报
- 没箭头的卡片拼贴
- 完整状态机图

## 3. 内容抽取顺序

制作流程图时，优先读取：

1. 目标 skill 的 `SKILL.md`
2. 被 `SKILL.md` 明确引用的 `references/`
3. 被 `SKILL.md` 明确引用的 `scripts/`
4. 被 `SKILL.md` 明确引用的 `assets/`
5. 必要时补读 `README.md`

## 4. 先做内容稿，再做显示稿

先整理流程图内容稿，至少包含：

- 标题
- 阶段
- 步骤
- 主干关系
- 箭头连接关系
- 每步来源

不要跳过内容稿，直接在 HTML 里临时拼一版。

## 5. 步骤数据规则

每个步骤建议至少包含：

- `id`
- `phase_id`
- `order`
- `track`
- `title`
- `source_kind`
- `source_dir`
- `source_anchor`
- `next`

规则：

- 一步只说一个动作或一个判断
- 标题要像普通人能理解的过程动作
- 不要把多个动作揉进一个标题
- `track` 默认使用 `main`
- `next` 默认只写“展示层需要画出来的关系”

### 5.1 `next` 的默认含义

`next` 不是“所有逻辑上成立的后续步骤集合”，而是：

`这张默认学习图上，应该明确画出来的下一步`

因此默认采用：

- `next = 展示层关系`
- “逻辑上也成立，但不值得单独画线”的关系，不进入 `next`
- 这类信息优先写进当前步骤的 `purpose`、标题或说明文字里

示例：

- 可以写：`01 -> 02 -> 03`
- 不默认写：`01 -> 02`、`01 -> 03`、`02 -> 03` 全部并存

如果 `01` 在某些情况下可以直接进入 `03`，默认做法是：

- 保留 `01 -> 02 -> 03` 作为图上的主线
- 在 `01` 或 `02` 的文案里补一句“信息完整时可直接进入下一步”

而不是额外再画一根 `01 -> 03` 的跳步线。

## 6. 主干优先判断

### 主干

主干步骤用于表达：

- 用户输入要求
- Agent 是否继续追问
- 关键分叉
- 关键约束
- 交付出口

经验规则：

- 如果不写进 `SKILL.md`，Agent 很容易做错，这类步骤优先进入主干
- 默认对外展示时，读者应该沿着单主干把流程读完

### 特例分支

只有在极少数情况下，才需要单独保留分支步骤，例如：

- 用户真的要做一个显式分支决策
- 流程真的会稳定分成两条并行路径
- 不单独画出来就会误解流程骨架

默认不要把这些内容单独升成分支：

- 补充理解
- 素材依赖
- 模板依赖
- 参考文档依赖
- 调试或校验辅助动作

它们优先并入主干步骤的文案说明里。

## 7. 展示优先级规则

默认流程图采用“单主干优先”原则。

### 7.1 默认画法

默认只强调：

- 一条清晰可读的主干
- 足够解释流程的箭头

### 7.2 不默认画的关系

以下关系默认不单独画线：

- 同阶段的跳步线
- 只用于表达“有时可以省略”的可跳过关系
- 已经能从文案读懂的补充关系
- 不是关键决策点的并行说明
- 仅为了保留 `branch` 概念而拆出来的步骤

### 7.3 什么时候才升级为显式线

只有在下面这些情况，才值得单独升级为分支画法：

- 用户真的需要做一个分支决策
- 流程真的会分成两条并行处理路径
- 某条辅助路径不单独画出来就会误解
- 校验或回收步骤本身就是这个 Skill 的关键卖点

## 8. 来源分工规则

### `SKILL.md` 应该强调什么

`SKILL.md` 负责：

- 主流程约束
- 关键判断
- 关键顺序
- 交付边界

### `references/` 应该承载什么

`references/` 负责：

- 长说明
- checklist
- screenshot 规范
- layout 规则
- 提示词库

### `assets/` 应该承载什么

`assets/` 负责：

- 模板文件
- 静态骨架
- 可直接复用的页面资源

### `scripts/` 应该承载什么

`scripts/` 负责：

- 渲染
- 生成
- 校验
- 布局固化

### `README.md` 应该承载什么

`README.md` 只负责补充说明，不作为主约束来源。

## 9. 阶段命名规则

阶段表达“流程阶段”，不要表达文件类型。
推荐优先用过程语义命名，例如：

- 用户输入
- Agent 接单
- 需求定型
- 模板起稿
- 内容落位
- 配图处理
- 校验与交付

## 10. 箭头规则

箭头不是装饰，是流程顺序的唯一显式表达。

必须做到：

- 每个关键步骤之间都有清晰箭头
- 箭头方向符合真实流程
- 读者一眼能看出先后顺序
- 箭头只表达流程，不表达归属
- 箭头数量服从“读者先看懂主干”这个目标

不要出现：

- 只有卡片没有箭头
- 箭头只是装饰线
- 为了省事直接删掉箭头
- 为了逻辑完整，把读者不需要立刻看见的关系全画出来

## 11. 输出建议

默认中级模块里展示流程图成品，并直接进入 merged-preview.html。
如需保留中间产物或单独页，可额外输出：

- `*-process-flow.html`
- `process-flow-data.json`

但它们不是默认对外交付主文件。

## 12. 质检清单

交付前至少检查：

- 流程是否完整
- 阶段是否语义清晰
- 是否保持了单主干优先
- 步骤是否按真实执行顺序组织
- 每步是否有来源
- 箭头是否清晰
- 是否把该写在 `SKILL.md` 的核心约束误拆成了分支
- 是否把只属于 `references/assets/scripts` 的补充信息误放进了主线
- 是否存在其实可以写进文案、却被额外画成跳步线的关系

## 13. 最小闭环

至少做到：

1. 先整理流程内容稿
2. 给每一步挂真实来源
3. 先按单主干整理顺序
4. 只把默认展示需要的关系写进 `next`
5. 在成品里把箭头画清楚
6. 输出可直接展示的流程图 HTML

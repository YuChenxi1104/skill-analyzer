# 结构图规则

本文件只负责说明“结构图本身应该展示什么，以及节点文案应该怎么写”。

它不负责：
- 定义 HTML 页面的布局和交互
- 替代现有模板或生成脚本
- 指导 Agent 从零设计页面

页面呈现规则请看 `structure-slide-html.md`。

## 目标

当你要向非技术用户解释一个具体 Skill 的文件结构时，使用本规则生成“结构图内容”。

这里的结构图不是抽象架构图，而是一张面向真实文件夹的导览图。它要帮助用户快速看懂：
- 这个 Skill 大致由几块组成
- 每一块里放了什么代表文件
- 这些文件分别是什么形式
- 这些文件分别是干什么的

## 核心原则

一个好的结构图，只回答一个问题：

`这个 Skill 文件夹里到底装了什么？`

优先级顺序是：
- 先可识别
- 再可理解
- 最后才是尽量完整

不要为了“全”把图塞成目录树导出图。

## 结构图应该展示什么

结构图默认展示四层信息：
- Skill 根目录
- 顶层模块
- 每个模块下的代表文件
- 每个代表文件的类型和用途

结构图不应该承担这些任务：
- 解释运行流程
- 解释代码内部逻辑

## 默认阅读顺序

用户看结构图时，应该能自然按这个顺序理解：
1. 先看到根目录
2. 再看到顶层模块
3. 再看到代表文件
4. 再看到文件类型
5. 最后看到文件用途

## 三层粒度

### 第一层：总览图
只回答：

`这个 Skill 由哪几部分组成？`

展示内容：
- 根目录
- 4 到 6 个顶层模块
- 每个模块一句角色说明

### 第二层：局部展开图
只回答：

`这一部分里面到底装了什么？`

展示内容：
- 被展开的模块
- 3 到 5 个代表文件
- 每个文件的类型和用途

### 第三层：单页展示用结构图
只回答：

`怎样在一页里同时保留全貌和细节？`

这一层只定义“图的内容仍然是结构图”，不定义具体交互。  
交互规则放在 `structure-slide-html.md`。

## 顶层模块选择规则

优先展示对理解最有帮助的模块，通常是：
- `SKILL.md`
- `assets/`
- `scripts/`
- `references/`
- `outputs/`

如果目标 Skill 没有这些标准目录，就按真实结构选“职责最清楚的几块”。

顶层模块推荐数量：
- 最少 3 块
- 最多 6 块

超过 6 块时，优先合并次要区域或拆成多张图。

## 代表文件选择规则

永远优先展示“代表文件”，而不是“全部文件”。

每个模块通常展示 3 到 5 个代表文件即可。

代表文件优先从这些类型里选：
- 入口文件
- 核心逻辑文件
- 样式文件
- 数据文件
- 说明文件
- 校验脚本
- 图片或素材示例

跳过这些文件：
- 高度重复的同类文件
- 只有实现细节价值、没有讲解价值的底层文件
- 对初学者理解帮助很低的支撑文件

## 文件标注规则

每个文件节点都应该同时表达三件事：
1. 文件名
2. 文件类型
3. 文件用途

推荐格式：

`文件名 - 类型 - 白话用途`

示例：
- `SKILL.md - Markdown - Skill 的主说明`
- `index.html - HTML - 页面模板`
- `app.js - JavaScript - 交互逻辑`
- `plan.json - JSON - 结构化计划数据`
- `validate_plan.py - Python - 检查计划是否合法`
- `home-bg.webp - 图片 - 页面背景素材`

## 节点文案压缩规则

结构图中的说明文字必须是“图注式短句”，不能直接搬运原文长句。

要求：
- `label` 只写文件名或目录名
- `type` 只写类别，不写解释句
- `note` 默认只写一句短说明
- `note` 只保留“作用”或“用途”，不要同时塞入背景、流程、条件、例外
- `note` 优先写成结果导向的短句，而不是过程化长句

不允许：
- 直接复制 `SKILL.md` 中的一整句说明放入节点下方
- 在一个节点说明里同时解释“是什么、为什么、何时触发、如何执行”
- 用段落式语言充当图注
- 用作者旁白、过程化语言或元叙事语言充当说明

## 节点文案长度预算

为保证结构图稳定显示，默认采用以下长度预算：

- `label` 尽量控制在 18 个中文字符以内
- `type` 尽量控制在 12 个中文字符以内
- `note` 尽量控制在 24 个中文字符以内

如果超出预算，优先处理顺序是：
1. 先压缩文案
2. 再调整节点间距和画布尺寸
3. 最后才考虑局部放宽显示空间

不要把完整原句直接塞进结构图，让 HTML 自己兜底。

## 文案与布局的边界

这里负责的是：
- 选哪些节点
- 每个节点的 `label / type / note` 写什么
- 把 `note` 压成足够短、足够清楚的图注式短句

这里不负责的是：
- 临时把某个说明框单独挪远一点
- 让某个节点通过特殊间距“看起来不撞”
- 用例外布局去补文案没有压缩好的问题

如果结构图开始拥挤，默认顺序应该是：
1. 先继续压缩 `note`
2. 再让结构数据增加画布空间、层级间距和同层留白
3. 最后交给 HTML 模板按固定规则稳定渲染

这里还要明确一条：
- 不要在内容层预设“某个子节点天生就该单独掉到下一排”
- 同组子节点默认先按单排理解和组织
- 只有当“子节点框 + 自己的说明框”整体占位在同排确实放不下时，才允许进入分层

换句话说：
- 内容层负责把节点说清楚、说短
- 不负责通过手工分层来掩盖布局问题
- 是否需要上下分层，应由后续布局阶段根据真实占位来判断

## 中英文混排规则

结构图中的英文文件名、扩展名、配置名和路径片段，应尽量保持完整表达。

要求：
- `openai.yaml`、`index.html`、`render_structure_slide.py` 这类词应作为完整信息使用
- 不要为了缩短文案，把关键文件名拆坏或改写得失真
- 中文说明可以压缩，但英文标识尽量保持原样

## 面向 AI 小白的表达规则

可以保留技术词，但必须补一句白话角色说明。

推荐说法：
- `SKILL.md - 入口说明`
- `assets - 模板和素材`
- `scripts - 自动化工具`
- `references - 补充规则`
- `outputs - 最终结果`

不要只写目录名本身，不解释它的作用。

如果某个文件类型对用户可能陌生，就补一句白话解释：
- `JSON - 结构化数据文件`
- `Python 脚本 - 校验工具`
- `SVG - 图标或矢量图`

## 视觉层级规则

层级必须一眼可见，不能靠用户自己猜。

推荐层级：
- 根目录最大
- 顶层模块次之
- 代表文件更小
- 类型和用途最轻

要让用户自然感觉到：
- 文件夹是容器
- 文件从属于容器
- 说明文字是辅助理解，不是主角

## 信息密度规则

结构图首先要“看得清”，其次才是“装得下”。

推荐控制方式：
- 1 个根节点
- 4 到 6 个顶层模块
- 每个模块 3 到 5 个代表文件
- 默认不要超过 3 级阅读链路

如果必须继续往下展示一层，也要保证：
- 不遮挡
- 不靠极小字号硬塞
- 仍能区分主次

## 什么时候拆成多张图

出现以下任一情况时，应该拆成多张图：
- 总览图已经不可读
- 顶层重要分支超过 6 个
- 某个模块明显比其他模块更值得单独展开
- 某个模块的代表文件本身就足以单独成页

推荐拆分顺序：
1. 总览图
2. `assets` 展开图
3. `scripts` 展开图
4. `outputs` 或最终结果图

## 推荐讲解顺序

给非技术用户讲结构图时，按这个顺序解释：
1. 先指出整个 Skill 文件夹
2. 再指出主要模块
3. 再解释每个模块负责什么
4. 再举代表文件
5. 再说明这些文件是什么类型
6. 最后说明用户最终会拿到什么

不要一上来逐个念文件名。

## `fitness-island` 示例锚点

拆解 `fitness-island` 时，优先展示这些元素：
- `SKILL.md`
- `assets/index.html`
- `assets/app.js`
- `assets/styles.css`
- `assets/plan.example.json`
- `assets/animal-island/`
- `scripts/init_project.sh`
- `scripts/validate_plan.py`
- `scripts/verify_assets.py`
- `references/plan-json.md`
- `outputs/fitness-plan.md`
- `outputs/plan.json`
- `outputs/index.html`

解释时优先用角色化语言：
- `SKILL.md`：主说明
- `assets`：模板和素材
- `scripts`：初始化和校验工具
- `references`：补充规则和格式说明
- `outputs`：用户最终拿到的结果

## 输出检查清单

完成结构图内容前，确认这些点：
- 根目录清晰可见
- 每个顶层模块都有角色说明
- 展示的是代表文件，不是抽象空话
- 每个文件都带有类型和用途
- 节点说明是短句，不是原文整句
- 英文文件名和关键标识保持完整
- 没有把布局问题甩给 HTML 临场兜底
- 图在投屏或一页阅读场景下仍可读
- 没有试图一次性塞下全部目录
- 细节只在需要时展开
- 标签对非技术用户是可理解的

## 反模式

不要这样做：
- 在一页里塞完整目录树
- 只列文件名，不写类型
- 只列文件名，不写用途
- 默认把所有文件看成同等重要
- 只写抽象目录名，不给具体例子
- 用结构图解释运行流程
- 把字号缩到看不清，只为了装下更多内容
- 把整句原文或长段说明直接放进节点下方
